# Nav-MCP trimmbar machen — reflektionsfreie Registrierung

Statusdokument für das Vorhaben, den MCP-Server (`Nav.Language.Mcp`, Assembly `nav.mcp`)
**getrimmt** ausliefern zu können. Trimming setzt voraus, dass keine trim-unsichere Reflection
mehr im Aufruf-Pfad liegt. Ziel dieses Plans: die zwei verbliebenen Reflection-Stellen durch
**statisch aufgelöste Registrierung** ersetzen und die Trim-Konfiguration einhängen.

## Ausgangslage (empirisch gemessen, 2026-07-14)

Ein Trim-Publish (`PublishTrimmed=true`, `TrimMode=partial`, Property **nur** am App-Projekt —
als globale CLI-Property vererbt sie sich sonst an die netstandard2.0-Referenzen und bricht mit
`NETSDK1124`) läuft bereits **fehlerfrei durch** (`EXIT=0`), sobald die Sprachversion lokal auf
C# 12+ angehoben ist. Größenvergleich, jeweils self-contained Single-File `win-x64`, Debug:

| Variante | Größe |
|---|---|
| Regulär (komprimiert, untrimmt) — wie `Publish-Mcp` | **37,8 MB** |
| Getrimmt + komprimiert, `partial` | **13,8 MB** (−63 %) |

Der Publish *baut* — die getrimmte exe ist damit aber **noch nicht** laufzeit-verifiziert: genau die
Reflection-Stellen sind es, die Trimming zur Laufzeit still brechen kann.

### Die zwei Reflection-Stellen

| # | Ort | Mechanik | Trim-Warnung | Eigentümer |
|---|---|---|---|---|
| **A** | `Nav.Language/SemanticAnalyzer/Analyzer.cs` (`GetInterfaceImplementationsFromAssembly<INavAnalyzer>`) | `Assembly.ExportedTypes` → Filter auf `INavAnalyzer` → `Activator.CreateInstance` (~50 Analyzer) | IL2026 **+ IL2075 + IL2072** | Unser Code |
| **B** | `Nav.Language.Mcp/Program.cs` (`WithToolsFromAssembly()`) | Reflection-Discovery der 14 `[McpServerToolType]` | IL2026 | MCP-SDK |

**A** ist die kritische Stelle: IL2075/IL2072 besagen, dass der Trimmer die Analyzer-Typen entfernen
darf (kein statischer Verweis) — die getrimmte exe könnte also **stumm Diagnostics verlieren**.

Zusätzlich (nur informativ, nicht Teil dieses Plans): `IL2104`-Rollups für `NLog`,
`NLog.Targets.Network`, `System.CodeDom` — reine Abhängigkeits-Assemblies.

## Weitere Voraussetzung: Sprachversion

Der MCP-SDK-Sourcegenerator (der die Tool-/JSON-Metadaten erzeugt) verlangt unter Trimming
**C# 12** und meldet sonst `error SYSLIB1102`. Das Repo ist projektweit auf `LangVersion 11.0`
gepinnt (wegen net472/netstandard2.0). Lokal im MCP-`.csproj` (net10.0-only) ist die Anhebung
auf `latest` unkritisch.

## Entscheidungen

- **Umfang: A + B1** (voll reflektionsfrei). A per eigenem Quellgenerator, B durch Umstieg auf die
  generische, trim-sichere SDK-Registrierung `WithTools<T>()`.
- **Vorgehen:** erst dieses Design-Doc, dann Step für Step mit Review + Build/Test + fertiger
  Commit-Message (kein Selbst-Commit).

## Design A — Analyzer-Registry-Generator

Strukturell ein vereinfachter Zwilling von `Nav.Visitor.SourceGenerator`: Typen der Compilation
finden → deterministischen, statischen Registrierungscode emittieren.

**Neues Generator-Projekt** `Build/SourceGenerators/Nav.Analyzer.SourceGenerator`
(netstandard2.0, `IIncrementalGenerator`), eingehängt wie Grammar/Visitor per
`ProjectReference … OutputItemType="Analyzer" ReferenceOutputAssembly="false"` in `Nav.Language.csproj`.

- **Prädikat:** nicht-abstrakte `ClassDeclarationSyntax`.
- **Transform:** Symbol ist `INavAnalyzer`-zuweisbar **und** hat einen zugänglichen, parameterlosen
  Konstruktor (heutige `Activator.CreateInstance(type)`-Invariante).
- **Emit** (`Analyzer.Registry.g.cs`): `partial class Analyzer` mit einer statisch verweisenden
  Fabrik — dadurch hält der Trimmer die Typen:

```csharp
// <auto-generated/>
namespace Pharmatechnik.Nav.Language.SemanticAnalyzer;
partial class Analyzer {
    internal static INavAnalyzer[] CreateAll() => new INavAnalyzer[] {
        new Nav0010CannotResolveTask0(),
        new Nav0011CannotResolveNode0(),
        // … alle Impls, Ordinal-sortiert
    };
}
```

**Handgeschriebener Teil** (`Analyzer.cs`) verliert Reflection, behält Lazy-Cache; `Analyzer` wird
`partial`:

```csharp
static partial class Analyzer {
    private static readonly Lazy<IList<INavAnalyzer>> _analyzers =
        new(() => CreateAll(), LazyThreadSafetyMode.PublicationOnly);
    public static IEnumerable<INavAnalyzer> GetAnalyzer() => _analyzers.Value;
}
```

Der Konsument `CodeGenerationUnitBuilder` (`Analyzer.GetAnalyzer()`) bleibt **unverändert**.
Snapshot-Test-Projekt analog zu `Nav.Visitor.SourceGenerator.Tests`.

**Nebengewinn:** kein Assembly-Scan/`Activator` beim ersten Analyze mehr (minimal schneller), und die
Analyzer-Liste ist zur Compile-Zeit sichtbar/prüfbar.

## Design B1 — MCP-Tools reflektionsfrei registrieren

Der trim-sichere SDK-Pfad ist `WithTools<T>()`; der ist **nicht** mit `[RequiresUnreferencedCode]`
markiert, weil `T` statisch bekannt ist und der SDK-Sourcegen die Metadaten dafür erzeugt.

Der frühere Blocker (`CS0718`, „statische Typen nicht als Typargument") betrifft nur die **Klasse**
als Typargument — **nicht** die Methoden. Umbau je Tool daher minimal:

- `public static class NavXxxTool` → `public sealed class NavXxxTool` (Methoden bleiben `static`).
- Die DI-Injektion von `NavMcpWorkspace` als **Methoden-Parameter** bleibt unverändert.
- `Program.cs`: `.WithToolsFromAssembly()` → 14× `.WithTools<Tools.NavXxxTool>()`.

14 triviale Klassendeklarations-Edits + eine Registrierungs-Umstellung.

## Trim-Konfiguration (`Nav.Language.Mcp.csproj`)

```xml
<PublishTrimmed>true</PublishTrimmed>
<TrimMode>partial</TrimMode>
<LangVersion>latest</LangVersion>
```

`PublishSingleFile`/RID/Kompression setzt weiterhin `Publish-Mcp.ps1` per Kommandozeile. Die
`PublishTrimmed`-Property gehört **ausschließlich** ins App-`.csproj` (nie als globale CLI-Property).
Der irreführende Kommentar „Kein PublishTrimmed …" in der csproj wird ersetzt.

## Offene Design-Punkte (vor bzw. während der Umsetzung zu klären)

1. **Reihenfolge.** Heute ist die Analyzer-Reihenfolge Metadaten-Reihenfolge (undefiniert).
   `DiagnosticsComputer` sortiert Diagnostics nach `Location.Start` (stabiler Sort) — Analyzer-Ordnung
   wirkt sich also **nur** bei Diagnostics an *identischer* Position aus (Tie-Break). Der Generator
   sortiert Ordinal; die Regression-/Diagnostics-Tests müssen bestätigen, dass sich keine
   Diagnostic-Reihenfolge verschiebt. Falls doch: sekundären, stabilen Sortierschlüssel definieren.
2. **Sichtbarkeit.** `ExportedTypes` findet heute nur **public** Typen; ein interner Analyzer würde
   ignoriert. Aktuell sind alle Analyzer `public` → praktisch neutral. Der Generator nimmt bewusst
   „alle nicht-abstrakten Impls" (korrekter); die Abweichung wird als kleine Verbesserung notiert.

## Step-Plan

1. **Generator-Grundgerüst A** — Projekt `Nav.Analyzer.SourceGenerator` + Snapshot-Test-Projekt,
   emittiert `Analyzer.CreateAll()`. Beide TFMs grün, Snapshot abgenommen.
2. **A einhängen + Reflection raus** — `Nav.Language.csproj` referenziert den Generator; `Analyzer.cs`
   auf `partial` + Lazy→`CreateAll()` umgestellt, `System.Reflection`-Import weg. Diagnostics-/
   Regression-Tests grün (Reihenfolge-Check gemäß offenem Punkt 1).
3. **B1 — Tools** — 14 Tool-Klassen `static`→`sealed`, `Program.cs` auf `WithTools<T>()`. Build grün,
   MCP-Tests (`Nav.Language.Mcp.Tests`) grün.
4. **Trim-Konfiguration** — csproj-Properties + Kommentar-Korrektur. Trim-Publish `EXIT=0` **ohne**
   IL2026/IL2075/IL2072 aus unserem Code (nur noch die IL2104-Dependency-Rollups).
5. **Abnahme** — stdio-Smoke gegen die **getrimmte** `deploy\mcp\nav.mcp.exe`: `nav_validate` **und**
   `nav_diagnostics` (damit die Analyzer real laufen und bewiesen ist, dass keiner weggetrimmt wurde).

## Abnahme-Kriterien

- Trim-Publish ohne IL20xx-Warnungen aus `nav.mcp`/`Pharmatechnik.Nav.Language`.
- Alle Tests (net472 via `nav test`, net10.0 via `dotnet test`) grün, inkl. `Nav.Language.Mcp.Tests`.
- Getrimmte exe liefert für eine bekannte fehlerhafte `.nav` dieselben Diagnostics wie die untrimmte
  (kein still fehlender Analyzer).

## Step-1-Handoff (turnkey — nahtloser Start in neuer Session)

Step 1 ist **eigenständig**: nur das neue Generator-Projekt + sein Snapshot-Test, beide grün. Der
Generator wird hier **noch nicht** von `Nav.Language` referenziert (das ist Step 2). Alles baut mit
`dotnet` (netstandard2.0 + net10.0) — **kein** MSBuild.exe nötig.

### Vorlage: alles kopiert vom bestehenden `Nav.Visitor.SourceGenerator`

Der neue Generator ist strukturell ein vereinfachter Zwilling. Referenzdateien zum Abschauen:
`Build/SourceGenerators/Nav.Visitor.SourceGenerator/{SymbolVisitorGenerator.cs, VisitorModel.cs}`
und `…/Nav.Visitor.SourceGenerator.Tests/{Shared/GeneratorTestBase.cs, Shared/SnapshotAssert.cs}`.

### Anzulegende Dateien

1. **`Build/SourceGenerators/Nav.Analyzer.SourceGenerator/Nav.Analyzer.SourceGenerator.csproj`** —
   identisch zum Visitor-Pendant; `SourceGenerator.props` liefert netstandard2.0, `IsRoslynComponent`,
   `Microsoft.CodeAnalysis.CSharp` **und** — via `Nav.CodeAnalysis.Shared.projitems` — den
   `SourceBuilder` (Namespace `Pharmatechnik.Nav.Language.CodeAnalysis.Shared`). Keine weitere Referenz nötig:

   ```xml
   <Project Sdk="Microsoft.NET.Sdk">
     <PropertyGroup>
       <RootNamespace>Pharmatechnik.Nav.Language.Analyzer.SourceGenerator</RootNamespace>
     </PropertyGroup>
     <Import Project="..\SourceGenerator.props" />
   </Project>
   ```

2. **`…/Nav.Analyzer.SourceGenerator/NavAnalyzerRegistryGenerator.cs`** — Skelett:

   ```csharp
   using System; using System.Collections.Immutable; using System.Linq; using System.Text;
   using Microsoft.CodeAnalysis;
   using Microsoft.CodeAnalysis.CSharp;
   using Microsoft.CodeAnalysis.CSharp.Syntax;
   using Microsoft.CodeAnalysis.Text;
   using Pharmatechnik.Nav.Language.CodeAnalysis.Shared;

   namespace Pharmatechnik.Nav.Language.Analyzer.SourceGenerator;

   [Generator(LanguageNames.CSharp)]
   public sealed class NavAnalyzerRegistryGenerator: IIncrementalGenerator {

       const string AnalyzerInterface = "Pharmatechnik.Nav.Language.SemanticAnalyzer.INavAnalyzer";

       public void Initialize(IncrementalGeneratorInitializationContext context) {
           var analyzers = context.SyntaxProvider
               .CreateSyntaxProvider(
                    predicate: static (node, _) => node is ClassDeclarationSyntax c &&
                                                   !c.Modifiers.Any(m => m.IsKind(SyntaxKind.AbstractKeyword)),
                    transform: static (ctx, _) => ToInfo(ctx))
               .Where(static x => x is not null)
               .Select(static (x, _) => x!)
               .Collect();
           context.RegisterSourceOutput(analyzers, static (spc, items) => Emit(spc, items));
       }

       static AnalyzerInfo? ToInfo(GeneratorSyntaxContext ctx) {
           if (ctx.SemanticModel.GetDeclaredSymbol(ctx.Node) is not INamedTypeSymbol s) return null;
           if (s.IsAbstract) return null;
           if (!s.AllInterfaces.Any(i => i.ToDisplayString() == AnalyzerInterface)) return null;
           var hasCtor = s.InstanceConstructors.Any(c => c.Parameters.IsEmpty &&
                                                         c.DeclaredAccessibility != Accessibility.Private);
           if (!hasCtor) return null;
           return new AnalyzerInfo(s.ToDisplayString(SymbolDisplayFormat.FullyQualifiedFormat)); // global::…
       }

       static void Emit(SourceProductionContext spc, ImmutableArray<AnalyzerInfo> items) {
           var ordered = items.Distinct().OrderBy(i => i.FullTypeName, StringComparer.Ordinal).ToImmutableArray();
           const string itf = "global::Pharmatechnik.Nav.Language.SemanticAnalyzer.INavAnalyzer";
           var sb = new SourceBuilder();
           sb.AppendHeader();
           sb.Namespace("Pharmatechnik.Nav.Language.SemanticAnalyzer", ns =>
               ns.Block("static partial class Analyzer", c =>
                   c.Block($"internal static {itf}[] CreateAll()", b => {
                       b.AppendLine($"return new {itf}[] {{");
                       using (b.Indent()) {
                           foreach (var a in ordered) b.AppendLine($"new {a.FullTypeName}(),");
                       }
                       b.AppendLine("};");
                   })));
           spc.AddSource("Analyzer.Registry.g.cs", SourceText.From(sb.ToString(), Encoding.UTF8));
       }
   }

   sealed record AnalyzerInfo(string FullTypeName);
   ```

3. **`…/Nav.Analyzer.SourceGenerator.Tests/Nav.Analyzer.SourceGenerator.Tests.csproj`** — Kopie des
   Visitor-Test-csproj (net10.0, NUnit + `Microsoft.NET.Test.Sdk` + `NUnit3TestAdapter`,
   `Compile Remove Snapshots\**`), `ProjectReference` auf den neuen Generator, RootNamespace anpassen.

4. **`…/Nav.Analyzer.SourceGenerator.Tests/Shared/{GeneratorTestBase.cs, SnapshotAssert.cs}`** — 1:1
   vom Visitor-Test kopieren, Namespace anpassen, `assemblyName` in `CreateCompilation` umbenennen.

5. **Testklasse + Snapshot** — der Generator arbeitet **semantisch**, die Test-Compilation referenziert
   nur `object`/`Enumerable`; der Test-Quelltext muss die Minimalbasen selbst mitbringen:

   ```csharp
   const string Source = """
       namespace Pharmatechnik.Nav.Language.SemanticAnalyzer {
           public interface INavAnalyzer {}
           public abstract class NavAnalyzer: INavAnalyzer {}
           public class Nav0011Foo: NavAnalyzer {}
           public class Nav0010Bar: NavAnalyzer {}
           public abstract class NavBaseIgnored: NavAnalyzer {} // abstrakt → NICHT gelistet
       }
       """;
   ```
   Erwartung: `CreateAll()` listet Ordinal-sortiert `new global::…Nav0010Bar()`, dann `…Nav0011Foo()`;
   die abstrakte Klasse fehlt. Snapshot nach
   `Snapshots/Expected/<TestKlasse>/<Name>.cs` legen (erster Lauf schreibt `Actual`, dann übernehmen).

### `.slnx`-Einträge

Zwei `<Project>`-Zeilen in den Ordner `/Build/SourceGenerators/` (aktuell Zeilen 59–66), **vor**
`Nav.AssemblyInfo.SourceGenerator` (alphabetisch „Ana" < „Ass"):

```xml
<Project Path="Build/SourceGenerators/Nav.Analyzer.SourceGenerator/Nav.Analyzer.SourceGenerator.csproj" />
<Project Path="Build/SourceGenerators/Nav.Analyzer.SourceGenerator.Tests/Nav.Analyzer.SourceGenerator.Tests.csproj" />
```

### Fallstricke (wichtig)

- **`static partial class Analyzer`** im Emit: das handgeschriebene `Analyzer` ist eine `static class`;
  alle `partial`-Teile müssen den `static`-Modifier tragen — sonst CS-Fehler in Step 2.
- **Positional-Records sind hier erlaubt.** Der Generator-Teilbaum nutzt sie (`VisitorModel.cs`) —
  bewusste lokale Ausnahme von der globalen „keine Primär-Konstruktor-Records"-Regel.
- **Voll qualifizierte Namen** (`global::…`) im Emit → unabhängig von `using`s im Zielprojekt.
- **Kein `CompilerVisibleProperty`** nötig (der Generator liest die Compilation, keine MSBuild-Properties).
